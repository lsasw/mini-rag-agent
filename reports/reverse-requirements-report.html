项目定位是一个用于学习 Python + FastAPI + Vue + LangChain + LlamaIndex 的 Mini RAG Agent。它不是完整企业知识库产品，而是一个端到端闭环样例：上传文档、解析切分、本地检索、构造 prompt、调用 LLM 或 mock、展示答案与引用来源，并提供 LlamaIndex CRUD 和检索练习。

当前实现适合教学、Demo 和本地验证。若要演进为生产级 AI 知识问答或内部 Copilot，核心改造点包括：真实向量库与 embedding、知识库权限、文档解析与异步索引、流式回答、评测集与观测体系、Agent 工具调用安全边界。

6. 需求点详情

REQ-001 上传并索引文档

高置信

业务目标：用户上传本地知识文件，系统解析并切分成可检索片段，为后续问答提供上下文。

前端实现

侧边栏 label.upload-button 内嵌 file input，限制 accept=".txt,.md,.pdf"，handleUpload 调用 uploadDocument(file)，上传中显示 loading 文案，成功后刷新文档列表。

后端实现

routes/documents.py 校验文件名和扩展名，ingest_upload 使用安全文件名写入 data/uploads，loader.load_text 解析 txt/md/pdf，split_text 切分，add_document 写入 data/index.json。

证据：frontend/src/App.vue:80、frontend/src/api.ts、backend/src/backend/routes/documents.py、backend/src/backend/rag/ingest.py、backend/src/backend/rag/loader.py、backend/src/backend/rag/splitter.py。

异常场景：无文件名返回 400；非 txt/md/pdf 返回 400；解析失败、写盘失败目前没有细分错误码；前端 catch 只显示通用上传失败。

REQ-002 查看和删除已上传文档

高置信

业务目标：用户能确认当前知识库包含哪些文档，并能删除不再需要的文档及其切片。

前端实现

refreshDocuments 加载文档；Documents 面板展示文件名、chunk 数、字符数；删除按钮调用 handleDelete(document.id)。

后端实现

list_documents 从 index.json 返回文档元数据；remove_document 同时删除 documents 字典和对应 chunks；不存在时返回 404。

证据：frontend/src/App.vue:67、frontend/src/App.vue:98、backend/src/backend/rag/store.py、backend/src/backend/routes/documents.py。

REQ-003 基于文档进行 RAG 问答

高置信

业务目标：用户输入自然语言问题，系统基于已上传文档检索上下文并生成答案。

前端实现

Chat 区输入框通过回车或发送按钮触发 handleSend，把用户消息加入本地消息流，调用 sendMessage(text, useAgent)，再渲染 assistant 回答、模式和来源。

后端实现

POST /api/chat 接收 ChatRequest。默认进入 answer_with_rag，调用 store.search 检索 top chunks，拼接 prompt，交给 complete。没有 OPENAI_API_KEY 时返回 mock；有 key 时调用 ChatOpenAI。

证据：frontend/src/App.vue:155、frontend/src/api.ts、backend/src/backend/routes/chat.py、backend/src/backend/rag/chain.py、backend/src/backend/rag/llm.py。

异常场景：无检索内容时 prompt 中写入“当前知识库没有检索到相关内容”；前端请求失败时提示检查后端服务。

REQ-004 展示引用来源和检索得分

高置信

业务目标：让用户知道答案依据来自哪个文档片段，提升可追溯性。

前端实现

消息气泡中 message.sources?.length 时展示 Sources，包含文件名、chunk index、score 和 preview。

后端实现

Source schema 包含 document_id、filename、chunk_index、score、preview；store.search 用关键词 Counter 余弦相似度计算得分。

证据：backend/src/backend/schemas.py、backend/src/backend/rag/store.py、frontend/src/App.vue Sources 渲染。

REQ-005 RAG/Agent 模式切换

中高置信

业务目标：让学习者体验普通 RAG 与简单 Agent 工具调用的差异。

前端实现

顶部 el-switch 绑定 useAgent，发送消息时作为 use_agent 传给后端。

后端实现

routes/chat.py 根据 use_agent 选择 answer_with_agent 或 answer_with_rag。Agent 识别“时间/time”和“计算/calc”，追加工具调用记录，再把 mode 从 rag: 替换为 agent:。

证据：frontend/src/App.vue:51、frontend/src/App.vue:164、backend/src/backend/schemas.py、backend/src/backend/rag/agent.py。

风险：calculate 使用受限字符集和空 builtins 的 eval，教学场景风险较低，但生产环境应替换为表达式解析库；工具调用不是由 LLM 规划，只是关键词触发。

REQ-006 至 REQ-008 LlamaIndex CRUD 与检索

高置信

业务目标：提供一个独立的 LlamaIndex 学习面板，练习分类、创建、查询、更新、删除和检索。

前端实现

右侧 LlamaIndex CRUD 区包含分类筛选、标题/分类/内容表单、列表、编辑/删除按钮和检索框。API 封装包括 listLlamaCategories、createLlamaDocument、searchLlamaDocuments 等。

后端实现

routes/llama.py 暴露 categories/documents/search。LlamaCrudStore 用 backend/data/llama_documents.json 持久化文档，用 VectorStoreIndex 和 MockEmbedding(embed_dim=64) 建内存索引，创建调用 insert，更新调用 update_ref_doc，删除调用 delete_ref_doc。

证据：frontend/src/App.vue LlamaIndex 面板、frontend/src/api.ts、backend/src/backend/routes/llama.py、backend/src/backend/llama/crud.py、backend/data/llama_documents.json。

限制：MockEmbedding 适合学习，不代表真实语义检索质量；内存索引在服务重启后通过 JSON 重建，但没有并发写保护。

REQ-009 本地联调与健康检查

高置信

业务目标：保证前后端本地开发可快速启动和排查基础连通性。

前端实现

Axios 默认 baseURL 为 http://127.0.0.1:8000/api，请求失败文案提示后端地址；Vite 默认用于本地开发。

后端实现

GET /api/health 返回 {"status":"ok"}；CORS 允许 localhost/127.0.0.1:5173/5174。

证据：frontend/src/api.ts、backend/src/backend/main.py、backend/src/backend/routes/health.py。

REQ-010 问答效果评测

开发需求

业务目标：用简单脚本快速判断 RAG 问答是否命中特定关键词，并观察平均延迟。

实现：backend/eval.py 定义两个 case，使用 httpx.AsyncClient 调用 POST /api/chat，检查 expected keywords，输出 accuracy 和 average latency。

限制：当前评测只检查关键词，不区分召回质量、引用准确性、幻觉、拒答质量和多轮上下文，不足以支撑生产上线。

8. 后端实现分析

维度现状缺口/建议
API 设计 资源路径清晰：/documents、/chat、/llama、/health；Pydantic response_model 明确。 没有统一错误响应模型、分页、请求 ID、API 版本化和 OpenAPI 描述补充。
文档处理 文件扩展名校验；PDF 用 pypdf；文本按 UTF-8 ignore 读取；上传文件名用 Path(...).name 防路径穿越。 没有文件大小/页数限制、重复文件策略、解析失败处理、恶意 PDF 防护、异步任务队列。
检索 自实现关键词 token Counter 和 cosine，结果写入 Source。 不是语义向量检索；中文分词非常粗糙；没有 metadata filter、召回评估、重排和混合检索。
LLM 调用 通过 OPENAI_API_KEY 开关真实模型和 mock；默认模型 gpt-4o-mini。 缺少超时/重试/成本控制/限流/日志脱敏；prompt 没有防注入策略；没有流式响应。
数据持久化 RAG 索引用 data/index.json；Llama 文档用 data/llama_documents.json。 JSON 文件并发写风险高；缺少备份、迁移、事务、索引重建任务和数据校验。
安全权限 本地学习项目未引入认证授权。 生产化必须增加登录、知识库权限、文档隔离、审计日志、上传内容安全检查和 API key 管理。
可观测性 有 /health 和简单评测脚本。 缺少结构化日志、检索命中日志、LLM latency/cost 指标、错误告警和 trace。

阶段 1：RAG 能力升级 从关键词 Demo 变成可用语义检索。 接 embedding 和向量库；保存 chunk metadata；支持重建索引；增加检索参数和引用定位。 固定问题集引用正确率明显高于关键词检索；重启后索引可恢复。
阶段 2：产品化交互 提升用户问答体验和管理效率。 流式回答；历史会话；反馈按钮；文档处理进度；知识库/分类管理；移动端可访问 Llama 面板。 首 token 延迟、用户反馈率、失败率可观测；移动端主流程可用。
阶段 3：安全与权限 支持多人/多知识库场景。 认证、文档 ACL、检索权限过滤、审计日志、API 限流、密钥管理。 越权测试通过；关键操作有审计记录；敏感配置不进入前端或日志。
阶段 4：AI Agent 与评测闭环 让 AI 不只回答，还能可靠辅助管理和操作。 工具调用框架、人类确认、评测平台、prompt/检索配置版本化、灰度发布。 工具调用可追踪可回滚；质量评测进入 CI 或发布流程。

12. 风险、缺口与待确认问题

主要风险

  • 前端可读性 中文文案疑似乱码，影响真实用户使用和演示质量。
  • 数据一致性 JSON 文件直接读写，没有锁、事务或并发控制。
  • 检索质量 关键词检索和 MockEmbedding 无法代表生产语义检索效果。
  • 安全权限 无认证、无知识库隔离、无上传内容安全策略。
  • AI 安全 Prompt 注入、幻觉、引用错误、成本和延迟均未纳入系统性控制。

待确认问题

  1. 项目下一阶段目标是继续作为教学项目，还是要演进为可部署的知识库产品？
  2. 真实使用场景中是否需要多用户、多知识库、团队权限和私有部署？
  3. 模型供应商倾向 OpenAI、Ollama、本地 bge，还是云厂商模型服务？
  4. 文档类型是否只需 txt/md/pdf，还是要扩展 Word、Excel、网页、飞书/Google Drive 等数据源？
  5. 是否需要保留 LlamaIndex CRUD 作为独立学习模块，或合并到主知识库管理能力？
